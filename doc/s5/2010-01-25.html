So what is this Cloud thing?

Cloud is an archtectural style more than a specific technology.
  • The logical conclusion of 20 years of Web-stuff: "Make it programmatic."
  • Service-oriented.
  • Full of abstract objects with varying representations. We can do things with these objects -- normally GET, PUT, and DELETE.
    • Lots of HTTP.
  • Locality-independence (eventually).

Leveraging the Lab

  • Yahoo!'s cloud initiative is focused on (customer-facing) properties.
  • SysArch needs to develop strategies to prepare Ops for the coming sea change.
  • "If you haven't written the program you don't know what you're talking about." - Andy van Dam via Dan Duchamp
  • The lab is the right place for this.
  • Need an environment unconstrained by Yahoo!'s legacy pile. Nothing is sacred.
  • Lab-facing services have a limited set of stakeholders and fewer dependencies than corporate and production systems. Furthermore, no assumption of production SLAs

Fundamental Architecture

  • Small, independent web services exposing uniform REST interfaces:
    • All resources (URIs) are nouns.
    • HTTP gives us the verbs: GET, HEAD, PUT, POST, DELETE, OPTIONS, etc...
    • Addressibility means connectedness: programmatic links are good.
  • Utilize HTTP to its fullest:
    • Meaning in HTTP request method and response status; (e.g. 401 Unauthorized)
    • Proxy services can be used to update two services from a single interface.

Process

  • Short iterative development cycles aimed at a specific and small set of tasks.
  • Releases are regular -- features are either completely integrated and documented or deferred to a later release.
  • Worst-case: roll-back to the last release.
  • Also, release:deployment != 1:1

Code

  • Developed and maintained by System Architecture.
  • The only constraints on an implementation are:
    • Completely documented
    • Testable and integrated
    • Reviewed to be sane and consistent

Book of Jersey

Documentation is critical.
  • Developed as an extension of the code -- tracked by Bugs and VCS.
  • Documentation is reviewed to be complete before release.
  • Short release cycles allow us to adapt our focus as business needs change and lessons are learned.
